# First Public Release Steps

这份文档是你第一次把 SkillClip 公开出去时，最实际的操作顺序。

## 1. 先检查仓库内容

先过一遍：

- [README.md](../README.md)
- [PUBLIC_RELEASE_CHECKLIST.md](./PUBLIC_RELEASE_CHECKLIST.md)

重点确认：

- 没有私有路径
- 没有真实 API Key
- 没有明显内部运营稿

## 2. 确认仓库首页

重点看：

- [README.md](../README.md)
- [README_EN.md](../README_EN.md)

如果要更顺，建议至少补：

- 1 张工作台截图
- 1 张 Whole Flow -> Skill 截图

## 3. 确认仓库基础文件

必须有：

- [LICENSE](../LICENSE)
- [CONTRIBUTING.md](../CONTRIBUTING.md)
- [ROADMAP.md](../ROADMAP.md)
- [PRIVACY.md](../PRIVACY.md)

## 4. 本地跑一遍最核心流程

至少测这几步：

- 保存 Prompt
- 保存 Whole Flow
- 生成 Skill
- 编辑 Workflow Prompt
- 应用 Skill 到输入框

## 5. 公开仓库

建议顺序：

1. 先把仓库设为 public
2. 立刻补 `About`
3. 补 `Topics`
4. 检查首页 README 展示是否正常

## 6. 发第一个 GitHub Release

建议：

1. 新建 `v0.1.0`
2. 说明它是本地优先原型
3. 说明已知限制
4. 放一个 zip 包

## 7. 再发首波介绍

建议发布渠道：

- X / Twitter
- 即刻 / 小红书 / 朋友圈
- 独立开发者社区
- AI 工具社区
- GitHub 相关群组

## 8. 首发时最值得强调的点

不是“功能很多”，而是：

- 它不是 Prompt Manager
- 它是 Skill Compiler
- 它把 AI 对话沉淀成可复用 SOP

## 9. 发出去之后最先看什么

公开后先盯这三类反馈：

- 哪个平台最容易失效
- Whole Flow 生成的 Skill 哪些地方最像“瞎编”
- 用户在哪一步最容易卡住

## 10. 当前最推荐的实际顺序

最简单的公开顺序：

1. 补 2 张截图
2. 公开 GitHub 仓库
3. 发第一个 release
4. 发第一波介绍
5. 收 issue
6. 继续迭代
